# 同步上游最新版并保留本地魔改：技术设计

## 1. 基线与边界

- 本地起点：实施开始时重新确认当前 `main` HEAD；规划时为 `85427b18c`。
- 旧上游基线：`22e509c1efb2260e1537c78684f1a5e9f053b75a`。
- 目标上游：固定为 `7c28993f6bd9e92616f3f578212577f8b7c40b45`。
- 兼容范围：只保证 9 组本地定制及其直接依赖；其他行为采用目标上游实现。
- 交付范围：独立集成分支、验证证据、升级与回滚说明；不部署生产、不推送远端、不合并 `main`。

## 2. 总体策略

在独立 worktree 中从当前 `main` 创建集成分支，将固定上游提交作为 merge parent 合入，保留本地历史与明确的上游来源。冲突处理遵循以下优先级：

1. 受保护的项目身份、品牌、许可证和归属信息不得改变。
2. 不涉及本地定制的文件采用目标上游版本。
3. 涉及本地定制的文件先保留上游新架构，再按行为契约重新接入定制。
4. 不为旧版普通管理/API 行为新增兼容层。
5. 无法归类到 9 组定制或明确项目元数据的业务差异，立即停止并回到规划确认。

```text
当前 main ───────────────┐
                        ├─ merge/语义适配 ─ 定制回归 ─ 补丁重建 ─ Docker 验证
上游 7c28993f ──────────┘
```

## 4. 关键数据流

### 4.1 API Key 自助与任务归属

```text
Bearer key
  -> TokenAuthReadOnly
  -> controller
  -> GORM transaction / task query
  -> users.quota + tokens.remain_quota + redemptions.status
  -> tasks.token_id（查询归属） + private_data.token_id（计费归属）
```

- 兑换必须在同一事务中完成钱包、key 额度、使用记录和兑换码核销。
- 新任务同时写两种 token 归属；历史任务不回填。
- 所有 JSON 编解码继续使用 `common` 封装。

### 4.2 Sora/Seedance 请求与计费

```text
解析可复用请求体
  -> 识别 seedance-asset（短路视频倍率）
  -> 通道与模型白名单守卫
  -> content 参考视频精确/双倍计费
  -> 顶层 Seedance 视频字段双倍计费
  -> 原始字段透传，仅替换映射后的 model
  -> 保留上游扩展响应字段并替换公开任务 ID
  -> 轮询状态归一化 / 结算 / 内容代理
```

- 同一请求不得重复叠加 004 与 007 的 `video_input`。
- 只有实际被当前通道转发的字段才能触发计费。
- 资产任务不得叠加 `seconds`、`size` 或参考视频倍率。
- 参考 URL 获取应优先复用目标上游的受保护 HTTP/SSRF 机制。

### 4.3 双前端锁屏

- 后端把同一 `FRONTEND_LOCK_PASSWORD` 安全 JSON 编码后注入两份 index。
- 共享的无 UI helper 负责读取配置、localStorage、TTL、指纹和密码比较。
- `default` 使用其 React 19/TypeScript 设计系统实现 gate；`classic` 使用 Semi UI gate。
- 锁屏只包裹前端渲染，不影响 `/api`、`/v1` 等非浏览器接口。
- 该能力仍明确标注为弱隐藏，不作为安全边界。

## 5. 数据库兼容与迁移

- 保留 SQLite、MySQL >= 5.7.8、PostgreSQL >= 9.6 兼容。
- 新增/保留 `tasks.token_id` 使用 GORM AutoMigrate 和普通索引，不使用单数据库专属类型。
- 上游 Token 列类型、订阅字段和新表迁移按目标实现保留；数据库专属 SQL 必须有对应分支。
- 验证包括空库启动和“当前版本建库并写入代表性数据 -> 新版本复用同一数据卷启动”的升级路径。
- 数据库迁移不可仅靠旧二进制回滚；生产手册要求升级前做一致性备份，失败时恢复数据库快照与旧镜像。

## 6. 补丁重建

当前补丁存在“可应用但不可编译”的假通过。新方案要求：

1. 以 `7c28993f` 为干净基准，按 `001` 到 `009` 顺序重建补丁。
2. 每个补丁只承载其代码、测试和必要登记差异；交叉文件按编号顺序形成连续状态。
3. `005` 收拢仍需保留的 CI、脚本、makefile、README 和 multipart 差异。
4. 默认 `PATCH_BASE_REF`、两份 README 和定制索引更新到新基准。
5. `make verify-patches` 除应用检查外，还必须在重放 worktree 中执行编译和定制回归。
6. 增加补丁覆盖审计，确保补丁拥有的源文件与最终集成树一致；`video_duration.go` 等新增源文件不得遗漏。

## 8. 提交与回滚

- 使用中文主题提交，按上游合并、后端定制、视频定制、双前端、补丁/验证分组。
- 每个阶段提交后打本地检查点或记录 SHA；失败可回退到前一阶段提交。
- 不执行 force push、远端 push、`reset --hard` 或删除用户分支。
- 最终只报告集成分支、worktree、提交列表和验证结果，由用户决定是否合并。

## 9. 停止条件

出现以下任一情况时停止继续扩展并回到规划：

- 发现未归类的本地业务功能或第 10 组隐含定制。
- 目标上游提交不可获取或与规划 SHA 不一致。
- 任何数据库升级会丢失已有代表性数据。
- 为保留定制必须修改受保护身份/归属信息。
- 补丁重放树与集成树无法建立一致映射。
- Docker 验证需要真实生产凭证或不可逆外部操作。
