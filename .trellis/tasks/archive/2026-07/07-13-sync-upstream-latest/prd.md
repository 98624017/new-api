# 同步上游最新版并保留本地魔改

## Goal

将当前长期落后于上游的本地 new-api 分支同步到 `https://github.com/QuantumNous/new-api` 的固定最新目标提交，同时保留并验证项目现有的本地定制功能，降低后续继续跟随上游的维护成本。

## User Value

- 获得上游最新的功能、修复和依赖更新。
- 本地已经投入使用的定制能力在升级后继续可用。
- 建立可审计、可验证、可回滚的同步过程，避免一次性合并掩盖功能回退。

## Confirmed Facts

- 当前仓库包含较多本地定制，并且与远端上游已存在较大版本差距。
- 指定上游仓库为 `https://github.com/QuantumNous/new-api`。
- 当前本地分支为 `main`，远端配置为本地派生仓库 `origin` 和官方仓库 `upstream`。
- 当前项目锁定的原版上游基线为 `22e509c1efb2260e1537c78684f1a5e9f053b75a`（`v0.12.11-1-g22e509c1e`）。
- 截至 2026-07-13，官方 `upstream/main` 为 `7c28993f6bd9e92616f3f578212577f8b7c40b45`；最新标签为 `v1.0.0-rc.21`（提交 `bde9b2f44887d34ec54799ae191d50f97914359e`），主线比该标签多 2 个提交，均属于“未定价模型”页面的修复。
- 相对共同基线，本地 `main` 有 34 个提交，上游最新主线有 533 个提交；本地差异涉及 155 个文件，其中包含业务定制、测试、维护工具、文档和 Trellis 元数据。
- 仓库已登记 9 组本地定制，均具备 `docs/customizations/NNN-*.md` 与 `patches/NNN-*.patch` 配对记录。
- 现有 9 个补丁可在锁定基线 `22e509c1` 上按顺序完整重放。
- 现有 `make verify-patches` 只验证补丁可应用，不会编译重放后的代码，因此存在假通过风险。
- 实测在旧基线重放全部 9 个补丁后，`relay/channel/task/sora` 无法编译：`004` 引用了 `video_duration.go` 中的函数和变量，但该新增文件没有被任何补丁收录。
- `.github/workflows/docker-image-alpha.yml` 与 `.github/workflows/docker-image-arm64.yml` 也存在未纳入现有补丁的本地维护差异，需要在 `005` 中重新归类或明确移除。
- 现有补丁无法直接重放到最新 `upstream/main`：`001-api-key-self-service.patch` 已在 `controller/token_test.go` 和 `model/redemption.go` 发生冲突。
- 无落盘合并模拟发现明确冲突面，包括 CI workflow 删除/修改冲突、`.gitignore`、`AGENTS.md`、`common/init.go`、`makefile`、`model/redemption.go`、部分测试文件，以及上游前端目录重组后 `FrontendLock` 文件的位置冲突。
- 上游已将前端拆分/迁移到 `web/classic` 等目录，本地前端锁定功能需要按新结构重新接入，而不是原路径机械套用。
- 上游最新版仅部分覆盖 `001-api-key-self-service` 所依赖的数据结构：任务私有计费数据已有 `TokenId`，但任务表没有用于按 key 查询的独立 `token_id` 列，也未提供本地 API Key 兑换与任务查询能力。
- 上游最新版的任务失败退款会直接恢复令牌额度，而本地 `002-task-refund-restore-token-quota` 使用环境变量控制且默认关闭；同步时必须明确保留本地开关语义，不能把上游行为误当成完全等价实现。
- 在上游最新版中未发现 `003` 金额脱敏、`004/007` 参考视频定制计费、`006` 前端锁、`008` Seedance 资产任务、`009` Sora unknown 状态兼容及 `005` 本地维护工具的等价实现。
- 从锁定基线到目标上游，数据库层涉及 42 个 `model/` 文件和约 5300 行新增，包含新表、新字段、订阅模型扩展、Token 列类型迁移及 ClickHouse 日志库支持。
- 上游生产迁移代码未发现主动删表操作，但应用启动会执行 GORM `AutoMigrate`，并对 MySQL/PostgreSQL 执行部分列类型变更；真实数据升级需要备份、迁移演练和三数据库兼容验证。
- 目标上游同时包含 `web/default`（新版 React/TypeScript 前端）与 `web/classic`（经典前端），Docker 镜像和 Go 二进制会同时构建、嵌入两套资源，并支持运行时切换。
- 当前本地 `006-frontend-lock` 只实现于旧前端路径；若仅迁移到 `web/classic`，切换到新版默认前端后会绕过锁屏。
- 当前主机 Docker Engine 29.1.3、Docker Compose 2.40.3 可用于本地验证。
- 仓库现有测试广泛使用本地 `httptest`/mock，并已有异步视频失败 mock、退款黑盒脚本和 fixture，可在不访问真实供应商的情况下验证关键计费、轮询和错误路径。
- 目标上游相对锁定基线重构了 62 个 `router/` 与 `controller/` 文件，新增权限、系统任务、流量统计等接口，并删除或调整部分旧管理端/内部接口；这些变化属于上游目标版本自身的行为演进。
- 项目要求本地定制代码、测试、说明文档和 `patches/*.patch` 保持一致，并通过 `make verify-patches`。
- 项目必须继续支持 SQLite、MySQL 和 PostgreSQL。
- 当前工作区已有与本任务无关的未提交修改，规划和后续执行不得覆盖或误纳入这些修改。

## Requirements

- 本次同步目标固定为官方 `upstream/main` 提交 `7c28993f6bd9e92616f3f578212577f8b7c40b45`，后续上游新增提交不自动扩大本任务范围。
- 现有 9 组已登记本地定制全部属于必须保留的行为契约，包括外部 API、配置开关、计费规则、错误处理、前端访问控制和维护流程。
- 允许基于上游最新架构重写定制实现，并删除已被上游等价覆盖的重复代码；验收以对外行为、配置语义和回归测试为准，不要求保持旧代码形态。
- 除 9 组定制涉及的接口和行为外，不额外维护当前旧版本的 API 兼容层；普通 Relay、管理端和内部接口均以目标上游提交 `7c28993f` 的行为为准。
- 某个上游接口只有在被 9 组定制直接依赖时，才需要为本地定制保留必要的兼容适配。
- 识别当前代码对应的上游基线、目标上游版本以及两者之间的版本跨度。
- 盘点本地定制的代码、测试、配置、数据库变更、文档和补丁文件，并建立可追踪映射。
- 制定分阶段同步策略，明确冲突处理原则、验证门槛、失败回滚点和提交边界。
- 审计相对旧上游基线的全部本地差异；任何未归入 9 组定制或明确的项目元数据/文档类别的业务代码，必须先归类再继续。
- 同步后保留经确认仍需维护的本地定制行为；不得擅自删除或弱化定制功能。
- 遵守项目关于 JSON 封装、三数据库兼容、前端 Bun、国际化和上游请求显式零值的约束。
- 受保护的项目身份、品牌、元数据和归属信息不得修改、删除或替换。
- 本任务交付同步后的代码、验证结果、数据库升级说明和回滚手册，不包含实际生产部署。
- 使用本地 Docker 环境进行充分验证，覆盖应用启动、数据库迁移、关键定制回归和主要 API/前端烟雾流程。
- 自动化验证不得使用真实 AI/视频供应商凭证或产生外部调用费用；关键上游交互统一使用本地 mock，另提供真实渠道人工验收清单。
- Docker 验证矩阵覆盖 SQLite、MySQL、PostgreSQL 三种主数据库，并分别连接 Redis。
- PostgreSQL 组合额外接入 ClickHouse 日志库，验证建表、日志写入和 TTL 配置。
- 同一构建镜像必须分别验证 `default`、`classic` 两套前端及锁屏行为。
- 本地 mock 黑盒验证覆盖视频参考计费、失败退款、unknown 状态轮询和 Seedance 资产任务链路。
- 实施必须从当前 `main` 提交创建独立集成分支，并在单独 worktree 中完成；不得把当前工作区未提交文件带入集成分支。
- 验证完成后结果保留在独立分支等待审核，不自动合并 `main`，也不推送任何远端。
- `006-frontend-lock` 必须同时保护新版 `default` 与经典 `classic` 两套前端，并共享一致的密码配置、解锁状态和公告展示语义。
- 在用户审核并批准完整规划前，不开始代码合并或实现。
